# Roadmap — ConfigAdmin

Документ фиксирует запланированные доработки после первой рабочей версии (MVP: выгрузка расширений и конфигураций через `1cv8.exe DESIGNER /DumpConfigToFiles`, WPF + консоль, SQLite, мастер-пароль).

Текущее состояние продукта описано в [README.md](README.md).

---

## 1. Выгрузка всех баз

**Сейчас:** команда `export-all` есть только в консоли.

**План:**
- Кнопка «Выгрузить все базы» на главном экране GUI.
- Прогресс по каждой базе, итоговый отчёт (успех / ошибки).
- Опционально: остановка очереди при критической ошибке или продолжение с частичным успехом (уточнить при реализации).

---

## 2. Удаление и сброс данных

**Сейчас:** удаление клиента/базы реализовано в `ProfileService`, но недоступно из GUI. Сброс «с нуля» — только вручную (удаление `%AppData%\ConfigAdmin`).

**План:**
- Удаление **базы** из GUI (с подтверждением).
- Удаление **клиента** (с базами или с проверкой «у клиента есть базы»).
- **Сброс профиля целиком** — очистка SQLite, vault, всех сохранённых настроек (аналог «обнулить программу»).
- Явное предупреждение, что выгруженные XML на диске не удаляются автоматически.

---

## 3. Клиент как отдельная сущность и дерево баз

**Сейчас:** клиент — запись в БД + combobox при добавлении базы; список баз плоский (`Клиент / База`).

**План:**
- **Редактирование клиента** (имя, каталог выгрузки, комментарий).
- Главный экран в виде **дерева:**
  - корень — клиент (с его настройками);
  - дочерние узлы — базы этого клиента.
- Действия из контекста узла: добавить базу, редактировать, выгрузить, удалить, логи.
- Каталог выгрузки задаётся на уровне **клиента**; базы наследуют `{ExportRoot}/{ClientName}/{BaseName}/...`.

---

## 4. Полноценный журнал (отдельно от выгрузок) — **выполнено**

**Реализовано:**
- `%AppData%\ConfigAdmin\logs\` — общий лог приложения (Serilog)
- `%AppData%\ConfigAdmin\runs\{Client}\{Base}\{runId}\` — meta JSON, `/Out`, `/DumpResult` per-step
- Meta JSON и артефакты **не** пишутся в `{ExportRoot}/{Client}/{Base}/`
- GUI: экран журнала с датой, базой, маскированной командой, длительностью, кодом, ошибкой, кнопками «Каталог выгрузки» / «Артефакты запуска», панелью шагов
- `/Out` и `/DumpResult` подключены в CLI-командах выгрузки

---

## 5. Формат выгрузки: XML (файлы) или архив

**Сейчас:** только выгрузка в каталог XML (`/DumpConfigToFiles`, формат Hierarchical / Plain в коде, в GUI — без выбора).

**План:**
- Настройка **на уровне базы:** «выгружать в XML (каталог)» или «выгружать в архив (.cf / .cfe?)».
- **Проверить в документации и на реальной базе**, какие CLI-команды 1С поддерживают для:
  - основной конфигурации;
  - расширений (отдельно и `-AllExtensions`).
- Возможные кандидаты (уточнить после проверки): `/DumpCfg`, `/DumpDBCfg`, выгрузка расширения в файл — зафиксировать точные параметры в отдельном разделе README после spike.
- Оркестратор: разные шаги/пути назначения в зависимости от формата; атомарная замена файла/каталога сохраняется.

---

## 6. Автообнаружение расширений и исключение патчей

**Сейчас:** имена расширений вводятся вручную в профиле базы или на экране выгрузки.

**План:**
- Получать **список расширений** из базы без ручного ввода (spike: CLI, COM, временная выгрузка, чтение метаданных — выбрать рабочий способ для пакетного режима).
- В UI: чекбоксы по найденным расширениям вместо/в дополнение к текстовому списку.
- **По умолчанию исключать патчи 1С** (эвристика по имени, типу или флагу из метаданных — уточнить после spike).
- Кнопка «Обновить список расширений» в карточке базы.

---

## Вне scope этого roadmap (на будущее)

- MCP-интеграция, расписание, checksum снимков.
- Архивирование предыдущих успешных выгрузок (`archive/`).
- Авто-блокировка vault при простое, экспорт/импорт профилей.
- CI/CD, интеграционные тесты с установленной 1С.
